# AGENTS.md

## 1. 项目定位

本仓库用于实现毕业设计项目：

《基于 AI 编程辅助工具的赣州市旅游文化智慧服务平台设计与实现》

项目目标：
1. 做成完整、可运行、可继续迭代的前后端项目。
2. 优先保证本地稳定跑通与演示效果。
3. AI 功能必须深度融合到旅游文化场景，不做纯装饰聊天框。
4. 代码结构必须清晰，便于编程基础一般的同学理解、维护和答辩讲解。

硬约束：
1. 优先保证可运行首版，不做过度抽象。
2. 优先保证真实联调，不做只有静态页面的假实现。
3. 优先保证功能闭环，不盲目堆砌复杂特性。

---

## 2. 仓库目录约定

仓库采用 monorepo 结构，目录固定如下：

- `client-web/`：前台游客端 Vue 3 项目
- `admin-web/`：后台管理端 Vue 3 项目
- `server/`：Node.js + Express 后端
- `sql/`：MySQL 建表、初始化、seed 脚本
- `docs/`：项目文档、接口文档、测试说明、演示说明
- `README.md`：根目录总说明文档

后端目录推荐约定：

- `server/src/config/`：配置项、环境变量读取
- `server/src/constants/`：常量
- `server/src/models/`：Sequelize 模型
- `server/src/routes/`：路由注册
- `server/src/controllers/`：控制器
- `server/src/services/`：业务服务
- `server/src/middlewares/`：中间件
- `server/src/utils/`：工具函数
- `server/src/prompts/`：AI 提示词模板
- `server/src/uploads/`：本轮骨架保留的上传目录占位

前端目录推荐约定：

- `src/router/`：路由
- `src/views/`：页面
- `src/components/`：公共组件
- `src/api/`：接口请求封装
- `src/stores/`：Pinia 状态管理
- `src/utils/`：工具函数
- `src/styles/`：全局样式

## 3. 开发顺序约定

必须严格按以下顺序推进，不允许跳步骤大面积并行开发：

1. 先冻结 PRD 实施计划、接口规范、数据库表结构、目录规范。
2. 再完成 MySQL 初始化 SQL 和后端基础框架。
3. 先打通后台登录和内容管理 CRUD。
4. 再完成前台首页、列表、详情等展示功能。
5. 最后接入 AI 智能问答、AI 行程推荐、AI 文案生成。
6. 所有核心功能完成后，再做联调、测试、演示优化、部署文档。

推荐的最小演示闭环顺序：

1. 管理员登录
2. 后台新增景点/文章
3. 前台展示景点/文章
4. AI 问答可引用数据库内容
5. AI 行程推荐可引用数据库内容
6. 后台 AI 文案生成可落日志
7. 后台查看 AI 日志

---

## 4. 不允许做的事情

1. 不允许擅自更换技术栈。
2. 不允许引入与 PRD 无关的复杂功能，例如普通用户系统、评论、点赞、支付、预约下单、多角色权限、地图导航。
3. 不允许为了“高级感”引入重型基础设施，例如向量数据库、消息队列、微服务拆分。
4. 不允许前端直接调用大模型 API，必须统一经过后端 AI service。
5. 不允许把用户问题裸传给模型，必须先做数据库召回并拼接上下文。
6. 不允许只改 Sequelize 模型而不同步 SQL 初始化脚本。
7. 不允许只做静态页面冒充联调完成。
8. 不允许为了抽象而抽象，首版禁止过度封装。
9. 不允许破坏统一接口返回结构。
10. 不允许随意修改已冻结的数据库字段和接口路径，除非同步更新文档和调用方。
11. 不允许使用破坏性 Git 命令清空用户已有改动。
12. 不允许引入难以维护的黑魔法式代码。

## 5. 代码风格和命名规范

### 通用规范

1. 默认使用清晰、直白、便于学生理解的写法。
2. 代码优先可读性，其次才是技巧性。
3. 每个文件职责单一，不写超长上帝文件。
4. 非必要不做复杂抽象。
5. 对关键逻辑添加简短注释，注释重点解释“为什么”，不是解释显而易见的语句。

### 前端命名

1. Vue 页面组件使用 PascalCase，例如 `ScenicListView.vue`。
2. 通用组件使用 PascalCase，例如 `PageBanner.vue`。
3. Pinia store 使用 `useXxxStore` 命名，例如 `useUserStore`。
4. 路由 path 使用 kebab-case，例如 `/red-culture`。
5. API 文件按模块命名，例如 `scenic.js`、`article.js`、`ai.js`。
6. 样式类名尽量语义化，不写杂乱无章的缩写。

### 后端命名

1. 路由、控制器、服务按模块拆分。
2. 文件名使用小写加中划线或按现有风格统一，例如 `scenic.controller.js`。
3. 数据表名使用复数下划线风格，例如 `scenic_spots`。
4. 数据库字段名使用 snake_case。
5. JS 变量与函数名使用 camelCase。
6. 常量使用大写下划线命名。

## 6. 接口返回规范

所有接口统一返回：

成功：
```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

失败：
```json
{
  "code": 500,
  "message": "错误信息",
  "data": null
}
```

约束：
1. 前台接口统一前缀为 `/api/front`。
2. 后台接口统一前缀为 `/api/admin`。
3. 分页列表接口建议返回 `list`、`total`、`page`、`pageSize`。
4. 不允许部分接口返回驼峰，部分接口返回其它结构，必须统一。
5. 状态切换、删除、创建、更新接口都必须返回统一结构。
6. token 校验失败时返回统一鉴权错误信息。

---

## 7. 数据库变更规范

1. `sql/schema.sql` 是当前数据库结构草案真源，后续任何表结构变更必须先更新该文件。
2. 修改数据库字段时，必须同步更新：
   - SQL 脚本
   - Sequelize 模型
   - 接口文档
   - 相关前端表单和展示字段
3. 首版尽量采用“新增字段优先，少做破坏性变更”的策略。
4. 每次改表都要检查 seed 数据是否仍然可用。
5. 所有核心表必须有基础索引，尤其是：
   - 分类字段
   - 状态字段
   - 推荐字段
   - 创建时间字段
6. AI 日志表、内容表、配置表必须保留，不能因为“暂时没用上”而删掉。
7. 删除字段前必须先确认前台、后台、AI prompt 是否依赖该字段。

## 8. AI 模块实现约束

AI 模块必须是“赣州旅游文化垂直助手”，而不是普通闲聊机器人。

### AI 问答约束

1. 必须先从 `scenic_spots` 和 `articles` 中做关键词模糊召回。
2. 首版召回方式以 `like` 或等价模糊检索为主，不引入向量数据库。
3. 只取前几条高相关内容拼接成上下文。
4. 必须将“用户问题 + 召回上下文 + 角色约束”一起构造成 prompt。
5. 必须记录 `ai_chat_logs`。

### AI 行程推荐约束

1. 必须结合 `days`、`preferences`、`departureArea`、`pace`、`extraRequirement` 组织 prompt。
2. 必须先从数据库筛选候选景点和文章，再交给模型生成结果。
3. 结果必须偏向可演示、可理解、可落地，而不是过于空泛。
4. 必须记录 `ai_trip_logs`。

### AI 文案生成约束

1. 后台文案生成只服务于内容运营场景。
2. 输入必须包含 `targetType`、`targetId`、`inputData`。
3. 输出内容至少覆盖简介、推荐语、攻略摘要中的核心部分。
4. 生成结果应允许管理员人工编辑后再保存。
5. 必须记录 `ai_copywriting_logs`。

### Prompt 管理约束

1. 所有 prompt 模板必须放在 `server/src/prompts/`。
2. 不允许把长 prompt 直接硬编码在控制器里。
3. 不同 AI 能力要拆分不同模板文件。
4. prompt 必须体现“赣州旅游文化”语境，避免通用旅游废话。

### 稳定性约束

1. AI 接口失败时，系统必须有友好兜底提示。
2. 模型超时、无 key、无匹配上下文等情况都必须可控处理。
3. AI 不是唯一主功能，不能因为 AI 异常导致整个平台不可演示。

## 9. 每轮修改后必须自检的事项

每次完成一个功能点后，必须至少自检以下内容：

1. 当前修改是否符合 PRD 范围，没有偷偷扩功能。
2. 当前修改是否破坏了统一接口返回格式。
3. 数据库字段、后端模型、前端表单字段是否一致。
4. 路由路径是否与既定清单一致。
5. 页面是否有空态、加载态、错误态的基本处理。
6. 是否存在明显的未导入、路径错误、命名错误。
7. AI 相关改动是否同步记录日志。
8. 环境变量是否补充到示例配置说明。
9. 新增模块是否放到了正确目录。
10. 是否仍然满足“首版先可运行，不做过度抽象”。

如果涉及联调，还要额外检查：

1. 后台新增的数据能否在前台看到。
2. 上传图片后访问路径是否正确。
3. token 失效时前端是否能正确跳回登录。
4. AI 页面提交后是否能在后台日志页看到记录。

---

## 10. 可运行首版优先原则

这是本仓库的最高优先级规则。

1. 先做最小可运行闭环，再做增强。
2. 先保证数据真实流动，再做视觉优化。
3. 先保证页面都能打开、接口都能返回、核心流程都能演示。
4. 首版允许适度朴素，但不允许不可运行。
5. 首版允许少量实现保守，但不允许把复杂问题无限后置。
6. 如果“更优雅的架构”和“更快做出可演示版本”冲突，优先后者。
7. 如果“更多功能”和“更稳演示”冲突，优先后者。

---

## 11. 面向初学者的实现原则

由于项目维护者编程基础仍在提升中，后续实现必须遵守：

1. 代码尽量写得直观。
2. 模块命名尽量贴近业务含义。
3. 关键流程要配简明注释。
4. 文档说明要能帮助后续复盘和论文撰写。
5. 避免一次性引入过多高级概念。
6. 出现取舍时，优先选择容易讲清楚、容易答辩、容易维护的方案。

---

## 12. 最终目标

本仓库的最终交付不是“炫技项目”，而是一个：

1. 能本地跑通
2. 前后台完整
3. AI 深度融合
4. 页面可展示
5. 论文好写
6. 答辩好讲
7. 可持续迭代

的毕业设计项目。
